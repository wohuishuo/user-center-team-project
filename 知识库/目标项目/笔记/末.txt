十二、代码优化 
1. 通用返回对象 
1）设计目的
问题背景：当前接口直接返回业务数据（如用户对象或列表），当出现异常或空数据时，前端无法区分是正常业务场景还是系统错误。
典型案例：
用户查询返回空对象：可能是用户不存在（正常）或系统异常（错误）
搜索返回空数组：可能是无匹配结果（正常）或强制返回空数组（异常）
注册返回-1：需要额外信息说明具体失败原因
核心目标：通过标准化响应结构，明确区分业务成功/失败状态，并携带补充信息。
2）基础结构设计
data字段：封装原始返回的业务数据（如用户对象、列表等）
code字段：业务状态码（数字或字符串），示例：
0表示成功
50001表示特定业务错误
message字段：执行结果描述信息（字符串），示例：
成功时："OK"
错误时："用户操作异常"
扩展建议：可增加description字段提供更详细的错误说明
3）与HTTP状态码的区别
HTTP状态码局限：
数量有限（如404/500等）
无法精确描述业务错误原因
与具体业务逻辑无强关联
业务状态码优势：
可自定义丰富状态（如40001/40002等）
直接关联业务场景
支持更精准的错误处理
4）代码实现

实现Serializable接口支持序列化

构造函数设计：
支持message参数默认值处理
支持构造函数重载

4. 前端返回封装 
1）前后端数据格式适配
后端封装结构：后端对返回值进行了统一封装，包含code、data、message等字段
前端适配需求：前端需要对应修改接收逻辑，从返回对象中提取data字段
原始问题：直接返回user对象的方式需要改为返回封装后的data对象
2）基础响应类定义
类名：BaseResponse（与后端对应）
泛型设计：使用泛型T表示data字段类型
字段组成：
code: number（状态码）
data: T（核心数据）
message: string（描述信息）
description: string（详细说明）
3）接口返回值改造
改造步骤：
将原返回值包装为BaseResponse的data字段
修改API.ts中接口的返回类型声明
调用处需要从res.data中获取实际数据
状态判断：通过code===0判断请求是否成功
错误处理：非零code时抛出异常，携带description作为错误信息
4）全局响应拦截器
目的：统一处理所有接口的响应，避免每个接口重复判断code
实现原理：
在请求库中配置response拦截器
拦截器自动解析response.body
判断code值并统一返回data或抛出异常
核心逻辑：
优势：前端业务代码无需关心响应封装层，直接获取数据
5）错误信息展示优化
错误传递：将后端返回的description作为错误message
前端展示：通过message.error()方法显示错误提示
示例场景：当星球编号过长时，前端会显示"星球编号过长"的提示
6）接口统一改造
改造范围：所有API接口（登录、注销、搜索用户等）
注意事项：
需要检查每个接口的返回数据结构
特殊返回值（如纯数字）需要单独处理
保持前后端字段命名一致
7）自定义请求类实现
必要性：避免修改框架生成的request文件
实现方式：
通过extend方法创建自定义request实例
配置请求/响应拦截器
导出供业务代码使用
最佳实践：将全局处理逻辑封装在独立文件中
5. 全局响应拦截器实现 
1）核心实现逻辑
HTTP状态码处理：拦截器需要处理HTTP状态码（如404），匹配对应的错误信息文本并抛出前端提示
业务状态码处理：当后端返回特定业务状态码（如4000参数错误/40100未登录）时进行特殊处理
数据结构规范：约定响应数据格式为{code, message, data}，其中code=0表示成功
2）代码结构设计
// 请求拦截器示例
request.interceptors.request.use(config => {
  console.log(`Request URL:${config.url}`)
  return config
})
// 响应拦截器核心逻辑
request.interceptors.response.use(response => {
  const res = response.data
  if (res.code === 0) {
    return res.data
  } else if (res.code === 40100) {
    // 未登录处理
    message.error('请先登录')
    history.push('/login')
    return Promise.reject(res)
  } else {
    message.error(res.message)
    return Promise.reject(res)
  }
})
3）关键实现细节
登录重定向：当检测到40100未登录状态码时，自动跳转登录页并携带来源页面路径
错误信息展示：通过message.error()统一展示后端返回的错误描述信息
成功数据处理：仅当code=0时返回data字段，其他情况均视为异常
4）常见问题解决
重复报错问题：需检查拦截器与组件内错误处理的逻辑冲突
数据获取异常：当接口返回null/undefined时需特殊处理
UMI兼容问题：注意与框架自带request的冲突，需要正确导入自定义拦截器
5）最佳实践建议
分层处理原则：通用错误在拦截器处理，特殊业务错误在组件内处理
状态码规范：建议与后端约定统一的错误码体系（如4xxx客户端错误，5xxx服务端错误）
调试技巧：在拦截器中添加console.log输出完整的请求/响应信息
6）注意事项
避免过度封装：特殊业务场景可能不适合使用全局拦截器
错误处理顺序：注意拦截器与组件内错误处理的执行顺序
响应数据结构：确保前后端对响应数据格式的理解完全一致
十三、前端全局响应处理器 
1. 应用场景
统一处理需求：当需要对接口响应进行统一处理时使用，例如从响应体中取出字段，或根据集中处理错误
典型场景：
权限校验（401未登录）
错误状态码处理（如403无权限）
优势：避免在每个接口请求中重复编写相同逻辑，提高代码复用性
2. 实现原则
类型收敛：处理的响应值必须能收敛为一类特定类型（如所有401响应都对应未登录场景）
修改隔离：后端数据结构变更时（如改为），只需修改拦截器一处即可
兼容性：原有接口函数无需修改，保持历史代码的稳定性
3. 具体实现方法
1）基础实现步骤
工具选择：根据项目使用的请求库选择对应实现方式
示例：umi-request可通过extend方法继承原有请求特性
文件组织：建议创建独立文件配置全局请求类
调用方式：在发送请求时使用封装好的全局请求类
2）不同库的实现参考
umi-request：
使用运行时配置中的拦截器功能
通过extend方法扩展基础请求类
axios：
参考官方文档的拦截器章节
创建axios实例时配置响应拦截器
其他库：查阅对应官方文档的拦截器/中间件相关章节
4. 注意事项
适用边界：仅适用于具有通用响应格式的场景
错误处理：需明确区分业务错误和系统错误的不同处理方式
调试建议：
保留原始响应日志便于问题排查
对特殊接口可配置局部绕过拦截器的选项
5. 典型代码结构示例
// 响应拦截器核心逻辑
responseInterceptor: (response) => {
  const { code, data } = response;
  if (code === 40100) {
    redirectToLogin();
    return Promise.reject('请先登录');
  }
  return data; // 自动解构响应数据
}
