1）多环境开发 
多环境概念

定义：指同一套项目代码在不同的阶段需要根据实际情况来调整配置并且部署到不同的机器上
典型环境：

测试环境
生产环境
为什么需要多环境

环境隔离：每个环境互不影响，避免未测试代码影响线上用户
阶段区分：
开发阶段：使用低配置机器
测试阶段：专用测试服务器
生产阶段：高配置服务器应对高并发
项目优化：
本地日志级别可设为debug，生产环境需隐藏敏感日志
精简依赖：开发依赖（如Lombok）不打包到生产环境
参数调整：如JVM堆内存参数在不同环境设置不同值
本地开发环境特点

访问限制：
localhost只能本机访问
内网IP（如192.168.x.x）仅限局域网访问
典型配置：
数据库连接地址为localhost
后端API地址为127.0.0.1
前端项目运行在本地端口
环境差异示例

依赖管理：
开发环境：包含Lombok等编译期工具
生产环境：仅保留运行时必需依赖
日志管理：
开发环境：输出详细debug日志
生产环境：只记录关键错误日志
安全考虑：
避免浏览器控制台暴露敏感日志
防止通过日志信息泄露服务器地址

2）静态化 

定义：为每个路由输出独立的index.html文件
配置方式：在配置文件中设置exportStatic: {}
效果对比：
开启静态化：每个路由目录下生成独立index.html
关闭静态化：仅生成单个index.html
注意事项：
静态化不支持变量路由场景
支付宝等特殊环境需要生成带.html后缀的文件
打包时需确保dist目录未被占用

文档参考：umi官方文档对静态化有详细说明，建议遇到问题时查阅
默认行为：新版本umi可能默认开启静态化功能
4. 静态化生成的目录 

静态化效果: 静态化后目录中仅保留index.html文件，其他路由文件都被转换为静态HTML文件
实现原理: 根据页面路由为每个路由生成对应的index.html静态文件
1）静态化配置

配置方法: 在umirc.js中配置export default { exportStatic: {} }
生成过程: 执行umi build命令后自动为每个路由生成HTML文件
限制条件:
不支持变量路由场景
支付宝等特殊容器环境需要特殊处理
2）静态化应用场景
服务端限制: 当后端服务器不支持根据路由动态跳转时
SPA问题: 传统单页应用依赖JS路由判断，静态化可解决无服务端支持的问题
生成机制: 将路由对应的组件预渲染为独立HTML文件

路由匹配: 每个路由对应一个静态HTML文件
访问方式: 直接通过文件路径访问对应路由内容
优势: 不依赖服务端路由配置，可直接部署到静态服务器
5. 应用案例 
1）例题:静态化网页访问

单页面应用特点：
整个网站实际上只加载一个index.html文件
页面跳转和组件样式改变都由预先加载的JavaScript控制
页面切换时不会真正刷新整个页面

传统路由问题：
当直接访问特定路由(如/user-login)时，服务器会尝试查找对应文件
由于服务器上只有index.html文件，会导致404错误
刷新页面时也会出现同样问题
静态化解决方案：
为每个路由生成独立的index.html文件
访问/user-login时，服务器会返回user-login/index.html
这样就能正确加载页面而不会出现404错误
部署策略选择：
动态路由：需要配置Nginx等服务器支持
静态化：适合不需要服务器端渲染的场景
删除静态文件后，对应路由会再次出现404错误

环境配置技巧：
配置文件可添加环境后缀区分不同环境
例如：config.dev.ts(开发环境)、config.prod.ts(生产环境)
公共配置放在无后缀的config.ts中
可通过环境变量让项目识别当前环境
部署建议：
部署前务必查阅框架官方文档
官方文档通常会提供性能优化建议
不同框架可能有不同的最佳部署方案
环境区分可以实现不同环境下的差异化配置

4. 自己编译安装方式 
1）安装准备
依赖检查：执行./configure命令会检查系统是否满足NGINX编译要求，包括检查PCRE等必要库是否存在
常见报错处理：若出现"rewrite module需要PCRE库"错误，需先执行yum install pcre安装依赖库
HTTPS支持：建议同时安装openssl库以支持未来可能的HTTPS配置，执行yum install openssl
2）编译安装流程
配置阶段：./configure命令会生成编译配置文件，包含NGINX二进制文件路径等参数
编译过程：make命令将C语言源码编译为二进制可执行文件，这是NGINX运行的核心文件
安装验证：编译完成后在objs目录下会生成绿色可执行文件（绿色表示具有执行权限）

修改配置文件

通过ps -ef | grep nginx确认进程用户已变更
2）部署总结
完整流程：打包→上传→解压→配置→权限设置→重载
常见坑点：
配置文件路径变更
目录权限不足
配置未及时重载
调试技巧：善用cat查看文件、ls检查目录、ps查看进程
14. 部署后端 
1）安装JAVA 
安装方式：使用yum命令快速安装OpenJDK 18，命令为yum install -y，其中-y参数表示自动确认安装
环境配置：通过yum安装的JAVA会自动配置环境变量，无需手动设置
验证方法：安装完成后使用java -version命令验证安装是否成功
2）安装Maven 
下载方式：
从Apache官网下载预编译的二进制包(bin)
使用curl -o命令直接下载到服务器
当下载速度过慢时，可选择本地上传方式
安装步骤：
解压下载的压缩包：tar -zxvf apache-maven-*.tar.gz
配置环境变量：编辑/etc/profile文件，在最后添加maven的bin目录路径
刷新配置：source /etc/profile
验证方法：执行mvn -v查看版本信息确认安装成功
3）项目代码提交与推送 
准备工作：确保服务器已安装git工具（如未安装可使用yum install -y git）

使用git commit -m "提交信息"提交更改
使用git push推送到远程仓库
注意事项：提交时需要正确配置git用户名和密码（与代码托管平台一致）
4）项目代码克隆与编译 
克隆项目：使用git clone <仓库地址>命令将项目下载到服务器
编译命令：
基本编译：mvn package
跳过测试：mvn package -DskipTests
编译过程：
首次编译较慢，因为需要从国外源下载依赖
后续编译会利用本地缓存，速度更快

5）添加可执行权限项目执行 
权限设置：使用chmod a+x <jar文件名>为jar包添加可执行权限
启动命令：
前台运行：java -jar <jar文件名> --spring.profiles.active=prod
后台运行：在命令前加nohup，结尾加&（如nohup java -jar ... &）
进程管理：
查看运行进程：jobs或ps -ef | grep java
查看端口占用：netstat -ntlp
终止进程：kill <进程ID>或前台运行时按Ctrl+C
环境配置：生产环境需使用prod配置文件（通过--spring.profiles.active=prod指定）

17. 容器技术部署方式 
1）容器技术概述
主流部署方式：容器技术是目前大公司主流的项目部署方式，绝大多数项目都采用容器化部署
核心优势：将项目环境（如JAVA、NGINX）和代码整体封装，避免重复安装配置
类比理解：容器相当于轻量级的"应用箱子"，包含运行所需的所有组件
2）Docker核心概念
镜像机制：
可将项目依赖环境（如JAVA、NGINX）和代码打包成镜像
支持镜像下载、制作和分发
有权限的用户可共享使用镜像
部署流程：
下载镜像 → 启动镜像
无需执行复杂的安装命令
技术类比：
类似软件安装包
类似操作系统镜像安装方式
3）宝塔面板部署实践
环境准备：
自动安装JAVA环境（通过java -version验证）
内置NGINX服务管理
站点部署流程：
添加站点（可直接使用服务器IP）

上传前端打包文件（dist目录内容）
自动完成NGINX配置映射
配置管理：
通过可视化界面修改NGINX配置
自动生成网站文件目录配置
效率对比：
相比原生部署方式更高效
适合个人项目快速部署
4）注意事项
权限控制：镜像分发需要严格的权限管理
环境隔离：容器提供独立运行环境，避免依赖冲突
学习曲线：需要掌握Dockerfile编写和镜像构建技术
适用场景：
企业级项目推荐容器化
个人快速开发可用宝塔简化流程
四、两个项目配置一个端口 
1. 多项目部署方案
多站点配置：通过添加新站点并绑定不同域名实现，每个域名独立对应一个项目
端口复用方案：同一端口可启动多个网站，通过Nginx的location配置实现路径分流
路径转发规则：
访问/a开头的URL时指向目录A的文件
访问/b开头的URL时指向目录B的文件
核心原理：Nginx在80端口启动后，根据请求路径动态转发到不同后端或文件位置
五、添加JAVA项目 
1. 项目部署准备

六、访问测试 
1. 端口配置问题排查
防火墙配置：8080端口默认未开放，需要手动在防火墙中开放TCP 8080端口才能正常访问
访问验证：访问121:8080/api应返回404状态码，若非404则说明存在问题
常见现象：访问响应缓慢可能是端口配置或网络问题导致
2. IP协议版本问题
问题发现：服务默认只监听IPv6端口，未启动IPv4端口导致无法访问
解决方案：
修改配置文件server.properties，添加server.address=0.0.0.0配置
重启服务使配置生效
验证方法：通过宝塔面板查看服务状态和项目日志确认配置是否生效
3. 宝塔面板相关配置
安全组配置：需要在宝塔安全设置中配置8080端口的白名单
文件管理：修改配置后需要重新打包项目并上传到服务器
服务管理：通过宝塔的"网站"模块可以重启服务，查看运行状态和日志
4. 问题排查经验总结
排查流程：
检查防火墙端口开放情况
验证IP协议版本配置
检查安全组/白名单设置
查看服务日志定位具体问题
建议验证：
在不安装宝塔的环境下测试项目启动情况
确认服务默认监听的IP协议版本
最终解决：通过配置白名单和修改server.address参数后问题解决
七、使用Docker部署 
1. Docker介绍及安装 
安装方式：提供两种主要安装方式：通过Docker官网直接安装和使用宝塔面板安装
官网安装特点：提供Download for Windows/Linux等版本，安装过程简单直接
宝塔安装步骤：
在宝塔面板搜索"Docker"
选择"Docker管理器"进行安装
推荐使用稳定版
验证安装：
方法1：通过宝塔提供的Docker管理器界面验证
方法2：在Linux终端执行docker -v命令，若显示版本号则安装成功
2. Docker File定义 
作用：用于定义构建Docker镜像的方法，包含项目依赖和启动流程
获取建议：通常不需要从零编写，可参考GitHub等平台上的同类项目
文件位置：建议放在项目根目录下
1）from 
功能：指定镜像依赖的基础镜像
示例：Spring Boot项目可基于包含Maven 3.5和Java 8的镜像
优势：避免手动安装和配置基础环境
2）watch 
功能：指定容器的工作目录
示例：WORKDIR /app表示后续操作将在/app目录下进行
作用：规范项目文件在容器中的存放位置
3）copy 
功能：将本地文件复制到容器中
典型操作：
复制pom.xml文件用于Maven构建
复制src目录存放项目源码
目的：在容器中重现本地开发环境
4）run 
功能：执行构建命令
示例：在Spring Boot项目中执行Maven打包命令
特点：这些命令与本地开发时使用的命令一致
5）cmd 
功能：定义容器启动时默认执行的命令
特点：
可在运行容器时覆盖
通常包含环境变量设置
示例：java -jar user-center-backend.jar启动Spring Boot应用
关联性：与本地运行项目的命令保持一致
3. 部署后端项目 
1）代码上传方式
Git克隆：推荐使用git clone命令将代码从仓库复制到服务器，比直接上传文件更规范
权限问题：腾讯云轻量服务器默认使用lighthouse用户，需通过sudo以root权限执行命令
目录操作：克隆前需确保目标目录不存在或为空，否则可能产生冲突
2）Docker镜像构建
构建命令：docker build -t 镜像名:版本号 . 其中.表示使用当前目录的Dockerfile
版本标签：通过-t参数给镜像打标签（如v0.1），便于区分不同版本的镜像
构建过程：分步执行Dockerfile中的指令，包括拉取基础镜像、设置工作目录、复制文件等
3）构建策略选择
服务器构建：在Dockerfile中直接包含构建命令（如maven打包），适合简单项目
本地构建：先在本地生成构建产物（如jar包），再复制到镜像，可避免服务器构建耗时
选择建议：根据项目复杂度选择，复杂项目推荐本地构建后上传成品
4. 前端Docker File 
1）基础结构
基础镜像：通常选择包含Nginx的镜像（如from nginx）作为前端运行环境
工作目录：通过WORKDIR指定容器内的工作路径（如/usr/share/nginx/html）
用户设置：可选步骤，通过RUN命令设置容器运行时使用的用户
2）配置文件处理
Nginx配置：将本地配置文件复制到容器的/etc/nginx/conf.d/目录覆盖默认配置
路由配置：必须添加try_files$uri$uri/ /index.html解决前端路由刷新404问题
目录结构：建议创建专门的docker目录存放构建所需配置文件
3）部署流程
代码打包：将构建好的静态文件复制到Nginx的默认访问目录
端口声明：EXPOSE 80仅作说明用途，实际不限制端口绑定
启动命令：通过CMD ["nginx", "-g", "daemon off;"]保持Nginx前台运行
4）注意事项
ENTRYPOINT vs CMD：ENTRYPOINT更灵活，允许运行时追加参数而不覆盖原命令
配置修改：应通过本地修改后重新构建镜像，而非直接修改容器内配置
版本控制：Dockerfile应与项目代码一起提交到版本控制系统
5. 部署前端项目 
1）克隆前端代码
克隆方式选择：建议使用HTTPS方式克隆前端项目，若服务器未配置SSH密钥，使用SSH方式会报错
项目定位：需要先找到前端项目在代码仓库中的具体位置，获取正确的项目地址
权限问题处理：若出现"permission deny"错误，需检查当前用户对目标目录的写入权限
2）构建前端镜像
构建命令：将后端构建命令中的"back end"改为"front end"即可用于前端构建
构建过程：首先会拉取NGINX基础镜像，然后逐步执行Dockerfile中定义的构建步骤
构建验证：可通过宝塔面板的"镜像管理"或命令行docker images查看已构建的镜像
6. 构建优化 
1）优化指标
关键指标：镜像大小和构建时间是两个最重要的优化方向
问题示例：Java后端项目构建耗时可能长达十几分钟，严重影响开发效率
优化目标：通过技术手段同时减少镜像体积和缩短构建时间
2）多阶段构建
核心思想：将构建过程拆分为多个阶段，每个阶段完成后丢弃不需要的组件
具体实施：
第一阶段：使用Maven环境完成Java项目打包
第二阶段：仅保留生成的jar包，丢弃Maven环境
优势：最终镜像不包含构建工具，体积显著减小，构建效率提高
3）其他优化方法
分包构建：将大型项目拆分为多个子模块分别构建
缓存利用：合理利用Docker构建缓存加速重复构建过程
基础镜像：选择更轻量级的基础镜像作为构建起点
