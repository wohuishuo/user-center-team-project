框架定位：数据访问层框架，基于MyBatis的增强工具

3）逻辑删除功能

实现原理：
实际执行update操作而非delete
通过标记字段（如deleted）标识数据状态
配置步骤：
全局配置：
设置逻辑删除字段名（如deleted）
配置已删除值（默认为1）和未删除值（默认为0）
实体类注解：
在字段上添加@TableLogic注解
注意事项：
查询时会自动过滤已删除数据
更新操作会防止修改已删除数据
适用于需要数据恢复的场景
频繁需要展示的数据建议使用状态字段而非逻辑删除

SQL表现：
删除：update user set deleted=1 where id=?
查询：select id,name from user where deleted=0
常见问题：
insert时需要处理默认值
删除接口自动填充功能可能失效
支持多种字段类型（推荐Integer或Boolean）

3. 用户逻辑删除配置 
1）逻辑删除概念

定义：通过字段状态标记替代物理删除的数据保护机制
特点：
数据保护：防止误删导致数据永久丢失
审计追踪：保留操作记录便于问题排查

2）MyBatis-Plus配置

配置项：

logic-delete-value：已删除状态值（默认为1）

实现原理：框架自动过滤已标记删除的数据
3）实体类注解

@TableLogic：
标注逻辑删除字段
示例：@TableLogic private Integer isDelete;
注意事项：
字段需与数据库列名对应
建议使用Integer类型而非Boolean

      logic-not-delete-value: 0

十四、问题定位与修复 
1. 报错分析与解决
注解缺失问题：首次报错原因是缺少@RequestBody注解，导致请求参数无法正确绑定
重启必要性：修改代码后必须重启服务才能生效，否则会持续报错
空值判断错误：出现空指针异常是因为将空值与整数直接比较，应增加非空校验
2. 数据库字段同步
字段同步问题：数据库新增字段后，实体类未同步更新导致查询结果异常
代码生成方案：使用代码生成工具重新生成实体类，确保与数据库结构一致
注解补充：必须添加@TableLogic注解才能使逻辑删除功能生效
3. 用户权限处理
登录态完善：需要在设置登录态时同步存储权限信息，避免后续获取为空
Session配置：在application.yml中添加session.timeout配置控制会话有效期（默认86400秒）
历史会话清理：修改权限字段后需要清除旧的登录会话，可通过重新登录覆盖
4. 数据脱敏优化
密码过滤：查询用户列表时需要过滤敏感字段如password
代码复用：创建getSafetyUser方法统一处理脱敏逻辑，避免重复代码
流式处理：使用Java8的stream().map()进行集合元素批量处理
// 安全用户处理方法示例

2）配置全局请求路径 
前缀配置方法
prefix方式：在APP.ts全局配置文件中定义请求前缀，如http://localhost:8080，所有接口请求会自动拼接该前缀
环境变量读取：可通过process.env获取当前环境变量，根据开发/生产环境动态切换请求地址
优势：代码集中管理，新开发者能快速定位请求地址，语义化清晰
替代方案对比
拦截器方案：通过请求拦截器动态修改URL，适合需要复杂处理的场景
baseURL方案：语义化更强但需要额外配置，框架已封装prefix时建议优先使用原生方案
环境区分逻辑：本地测试时请求测试地址，线上环境自动切换为生产地址

代理基本概念
正向代理：替客户端向服务器发送请求（如跑腿代购）
反向代理：替服务端接收请求并分发到多台服务器（如负载均衡）
跨域本质：浏览器地址（如8000端口）与后端地址（如8080端口）不同源即产生跨域
代理配置实践
配置要点：

代理目标地址需包含完整端口号

路径冲突：避免代理路径与前端路由路径重复
