# Redis

> **结论先行**:Redis 是一个**把数据放在内存里**的数据库,所以快。在用户中心,它干两件事:**存登录态**(让多台服务器共享会话)和**做缓存**(挡住重复的数据库查询)。

## 一句话

| 项 | 值 |
| --- | --- |
| 定位 | 内存数据库 / 缓存 |
| 本书版本 | Redis 7 |
| 在用户中心的角色 | 共享登录态 + 缓存 |

## 解决什么问题

**结论**:它解决两个问题——**多台服务器怎么共享登录状态**,和**怎么少查几次数据库**。

**根据**:
- 登录态如果存在某台服务器的内存里,部署多台时用户会"掉线"(见 [认证模块](/project/auth));存进 Redis,所有服务器都能读到同一份。
- 数据库在硬盘上,查询相对慢;把热点数据放 Redis(内存),下次直接命中,快上几个数量级。

**例子**:用户登录后,会话放进 Redis;无论下次请求被转到哪台后端,都能从 Redis 认出他。

## 依赖关系

**结论**:后端通过 Redis 客户端连它,它独立运行在自己的进程/容器里。

```
Spring Boot
  └─ 依赖 spring-boot-starter-data-redis
       └─ 通过网络连接 → Redis 服务(独立容器)
            ├─ 存:Spring Session 会话
            └─ 存:缓存数据
```

- **它依赖谁**:不依赖业务代码,自己就是一个独立服务。
- **谁依赖它**:后端的会话管理和缓存逻辑;在 [Docker Compose](/stack/docker) 里作为一个独立容器编排。

## 版本与生命周期

**结论**:用官方 `redis:7` 镜像,后端配上连接地址即可。

```yaml
# application.yml
spring:
  data:
    redis:
      host: redis      # docker compose 里的服务名
      port: 6379
  session:
    store-type: redis  # 让 Spring Session 把会话存进 Redis
```

::: warning 内存数据要想好"丢了怎么办"
Redis 在内存里,重启可能丢数据。当缓存用,丢了无所谓(回数据库重查);当登录态用,丢了用户得重新登录——这通常可接受。真正重要的数据,主库永远是 MySQL。
:::

## 在用户中心里的角色

**结论**:两个具体用途。

1. **共享登录态**:配合 Spring Session,把会话从"单机内存"挪到"共享 Redis",让后端能水平扩展;
2. **缓存**:比如把"按用户名查用户"的结果缓存一小段时间,挡住重复查询。
