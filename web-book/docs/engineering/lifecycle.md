# 软件生命周期

> **结论先行**:做项目不是"想到哪写到哪",而是走一条固定的路——**需求 → 设计 → 编码 → 测试 → 评审 → 部署 → 维护**。每一步都有产物,前一步没做好,后一步会加倍偿还。

## 一、为什么要有"流程"

**结论**:流程的价值,是**把最贵的错误挡在最便宜的阶段**。

**根据**:同一个错误,在需求阶段发现改一句话,在上线后发现可能要赔一场事故。越早发现越便宜——这就是"最左原则"。

**例子**:用户中心如果在需求阶段就写清"密码必须加密",就不会等到上线被脱库才追悔。

## 二、八个阶段,各干一件事

**结论**:企业开发流程是一条清晰的流水线,每一站交付一个产物给下一站。

```
需求分析 → 设计 → 技术选型 → 编码 → 测试 → 代码评审 → 部署 → 上线维护
```

| 阶段 | 干什么 | 产物 | 在用户中心里 |
| --- | --- | --- | --- |
| 需求分析 | 搞清做什么、为谁做 | 需求规格说明书 | [需求分析](/project/requirements) |
| 设计 | 把需求变成结构 | 系统/数据设计说明书 | [软件设计](/theory/03-design) · [数据库设计](/project/database) |
| 技术选型 | 定用什么技术 | 技术栈清单 | [技术全景图](/guide/tech-landscape) |
| 编码 | 把设计写成代码 | 可运行代码 | [认证模块](/project/auth) |
| 测试 | 验证功能正确 | 测试报告 | [测试](/project/testing) |
| 代码评审 | 把关质量与安全 | 评审意见 / PR | [代码评审](/engineering/code-review) |
| 部署 | 让代码跑在服务器上 | 可访问的服务 | [部署上线](/project/deployment) |
| 维护 | 修 bug、加功能 | 新版本 | 见下文 |

**根据**:每一站的"产物"正好是下一站的"输入"。需求文档喂给设计,设计喂给编码,环环相扣。

## 三、不是一条直线,而是会回头的环

**结论**:真实开发不是一次走完,而是**不断迭代**——测试发现问题会退回编码,上线后的需求会退回需求分析。

**根据**:没有人能一次想全。把"会回头"当成正常,而不是失败,团队才不会因为返工而互相指责。

**例子**:用户中心先做出"能登录"的最小版本,跑通后再迭代加"用户管理"、再加"用户校验"。先完成,再优化。

::: tip 维护占了软件一生的大头
软件上线只是开始。后续的纠错(修 bug)、适应(跟上环境变化)、完善(加功能)合称"维护",往往占整个生命周期成本的一大半。所以**好维护**本身就是设计目标——这又回到了[高内聚低耦合](/theory/03-design)。
:::

## 四、AI 时代,流程没变,做法变了

**结论**:八个阶段一个没少,但每一站现在都有 AI 参与——人定方向,AI 出力。

**例子**:

- 需求阶段:让 AI 帮你把零散想法整理成用例;
- 编码阶段:让 [Claude Code](/engineering/ai-workflow) 按设计生成骨架;
- 测试阶段:让 AI 起草测试用例;
- 评审阶段:让 AI 在 GitHub PR 上先审一轮。

**根据**:AI 改变的是"谁出力",没改变"该走哪些步"。跳过设计直接让 AI 写代码,照样会塌——流程是给人和 AI 共同的护栏。

## 本章小结

- **结论**:软件生命周期是一条会回头的流水线,早发现早便宜;
- **根据**:每一站的产物喂给下一站,维护占一生成本的大头;
- **例子**:用户中心先做"能登录"再迭代,正是"先完成再优化"。
