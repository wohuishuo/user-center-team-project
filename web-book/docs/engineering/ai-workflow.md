# AI 协作:Claude Code 工作流

> **结论先行**:2026 做项目,人的角色从"逐行写代码"变成"**设计 + 审校**"。Claude Code(简称 CC)负责生成骨架、写测试、改代码,你负责说清要什么、并守住质量闸门。这一章讲**具体怎么用**,以及哪些坑必须自己盯着。

## 一、CC 能干什么:三件事

**结论**:把 CC 当成一个执行力极强、但需要你给清指令和边界的队友。它主要帮你做三件事:

1. **生成骨架**:根据需求和数据库表,生成 Controller / Service / Mapper / 实体类一整套;
2. **写测试**:为某个方法起草单元测试,覆盖正常和边界情况;
3. **改代码 + 解释**:重构、修 bug,并说明为什么这么改。

**例子**:你说"按 user 表生成注册接口,密码用 BCrypt",它能直接产出可编译的一套代码,你只需审一遍逻辑和安全。

## 二、Hooks:给 AI 划红线(以及它的坑)

**结论**:Hooks 是你在 CC 执行动作**前后自动运行的检查脚本**,用来强制规则——但它只能挡住"它知道要挡的东西",挡不住它没考虑到的连带影响。

### Hooks 怎么用

Hooks 配置在 `.claude/settings.json`,常见用途:

- **保护关键函数**:某个核心函数不许改,在 `PreToolUse`(编辑前)钩子里检查 diff,命中就拒绝;
- **数据库保护**:执行 SQL 前用钩子拦截 `DROP` / `DELETE` 这类危险语句,要求二次确认;
- **提交前自动检查**:`Stop` 钩子里跑一遍 lint / 测试,不过就不收尾。

### ⚠️ 必须自己盯着的坑

**结论**:Hooks 锁住了"函数 A 不许改",但**挡不住因为别处改动导致 A 的调用方崩掉**。

**例子**(真实会发生):

```
你用 hook 锁住了 函数A,禁止 CC 修改它。
CC 去改 函数B,顺手调整了 函数A 接收的某个参数的来源。
但 函数C 也依赖 函数A 的那个参数 —— CC 视野里没看到 函数C。
结果:函数A 本身没动(hook 满意),但 函数C 开始报错。
```

**根据**:Hooks 是"局部规则",而 bug 常出在"全局连带"。AI 的上下文是有限的,它没把 `函数C` 读进来,就意识不到这条依赖链。

**怎么防**:

- 改完**自己跑一次完整测试 / 启动一次应用**,别只信 hook 通过;
- 关键依赖关系,写进 `CLAUDE.md` 让 CC 每次都看到;
- 让 CC 改动后**主动列出"谁调用了我刚改的东西"**,再逐个确认。

## 三、MCP:把外部工具接给 AI

**结论**:MCP(Model Context Protocol)是让 CC 直接操作外部系统的标准接口——接上数据库、浏览器、GitHub 等,CC 就能"亲手"查库、点页面、提 PR,而不只是写代码给你。

**例子**:

- 接上**数据库 MCP**:CC 能直接查表结构、跑只读查询来验证数据,不用你复制粘贴;
- 接上**浏览器 MCP**:CC 改完前端能自己打开页面、截图,确认按钮真的能点;
- 接上 **GitHub MCP**:CC 能读 issue、建 PR、看 CI 状态。

**根据**:没有 MCP,CC 只能"写"不能"做";有了 MCP,它能闭环验证自己的工作。

::: warning 给 AI 接外部系统,先想清权限
数据库 MCP 建议**只给只读账号**;涉及删除、转账、改权限的动作,永远让人来按最后一个按钮。
:::

## 四、和 GitHub 打通:让 AI 在 PR 里审代码

**结论**:把代码推到 GitHub 后,可以让 CC 在 **Pull Request** 上自动审查——检查功能是否正常、有没有引入 bug,像多了一个不知疲倦的评审员。

**典型流程**:

```
1. 本地让 CC 实现功能 → 自己跑测试通过
2. push 到一个 feature 分支,开一个 PR
3. CI(GitHub Actions)自动跑构建 + 测试
4. 让 CC 审这个 PR:读 diff → 指出风险 → 在 PR 上留评论
5. 你看人 + AI 两轮意见,改完再合并到 main
```

**根据**:代码评审是为了"防止带 bug 或安全问题的代码上线"。AI 评审快、覆盖广,但**最终拍板合并的还是人**——这守住了 [写作方法](/guide/writing-style) 里说的"不搞纵向压制、人来负责"的原则。

**例子**:PR 里加一句"@claude 检查这个登录接口有没有漏掉鉴权",CC 会读相关文件,回你一条具体评论,比如"`/api/user/delete` 没校验管理员角色"。

## 五、常用命令心法

**结论**:命令不用背全,记住"**说清目标 + 给够上下文 + 守住闸门**"这套心法,具体命令查文档即可。

| 你想做的事 | 心法 |
| --- | --- |
| 让 AI 写新功能 | 先给它需求 + 数据库表 + 相关已有代码的位置,再让它动手 |
| 让 AI 改 bug | 给它**报错信息原文** + 复现步骤,别只说"它坏了" |
| 防止它改坏东西 | 用 hook 锁关键文件 + 改完自己验证 |
| 让它验证自己的工作 | 接 MCP,让它跑测试 / 开页面 / 查库自证 |
| 团队协作 | 走 PR 流程,人 + AI 双重评审再合并 |

## 本章小结

- **结论**:AI 时代,你设计与审校,CC 执行;
- **根据**:Hooks 划红线但有盲区,MCP 让 AI 能闭环验证,GitHub PR 让 AI 当评审员;
- **例子**:锁了函数 A 却忘了函数 C 依赖它——这类全局连带,必须靠自己跑一遍来兜底。

> **最重要的一条**:无论 AI 多强,**最后按下"合并 / 删除 / 上线"按钮的,永远是人。**
