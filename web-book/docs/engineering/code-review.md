# 代码评审

> **结论先行**:代码评审是合并前的最后一道质量闸门,目的是**挡住带 bug 或安全隐患的代码上线**。评审看四件事:能不能跑、安不安全、好不好懂、有没有重复。

## 一、为什么要评审

**结论**:再细心的人也有盲区,多一双眼睛就少一个事故。

**根据**:[软件生命周期](/engineering/lifecycle) 讲过"错误越早发现越便宜"。评审发生在合并前,是上线前最便宜的一次拦截。

**例子**:作者没注意到删除接口漏了管理员校验,评审的人一眼看出来——一次越权事故就这样被挡在了 main 之外。

## 二、评审看什么:四个维度

**结论**:照这四条过一遍,大部分问题都能拦住。

| 维度 | 问自己 |
| --- | --- |
| **正确性** | 功能真的实现了吗?边界情况(空值、超长)处理了吗? |
| **安全性** | 有没有漏鉴权?密码/隐私有没有泄露?有没有越权? |
| **可读性** | 半年后别人看得懂吗?命名清楚吗? |
| **复用性** | 有没有复制粘贴的重复代码该提取? |

**例子**:用户中心里最该盯的是"安全性"那一栏——每个管理接口是否都先鉴权、每个返回是否都脱敏。

## 三、怎么提意见、怎么收意见

**结论**:提意见**对事不对人**,收意见**不带情绪**——这是 [写作方法](/guide/writing-style) 里"平等、不搞纵向压制"的直接应用。

**根据**:评审的目标是让代码更好,不是证明谁更厉害。把"你写错了"换成"我们这里是不是漏了鉴权",对方更容易接受,事情也推进得更快。

**例子**:

- ✅ "这个删除接口好像没校验角色,普通用户会不会能调?"
- ❌ "你怎么又忘了鉴权?"

::: tip 组长/资深也要被评审
不是"谁级别高谁的代码就不用看"。所有进 main 的代码一视同仁地过评审——这才是健康的团队。
:::

## 四、AI 也来评审一轮

**结论**:人评审之外,让 Claude 在 PR 上先扫一遍,覆盖更广。

**根据**:AI 快、不疲倦、能一次性检查很多机械性问题(漏鉴权、空指针风险);人则负责判断"这个设计合不合理"这类需要上下文的事。两者互补。

**例子**:见 [AI 协作工作流](/engineering/ai-workflow) 里"在 PR 里 @claude 检查接口鉴权"的做法。但记住——**最后点合并的还是人**。

## 本章小结

- **结论**:评审是合并前最后的质量闸门;
- **根据**:四个维度(正确/安全/可读/复用)+ 对事不对人;
- **例子**:用户中心最该盯安全;AI 扫机械问题,人判断设计,人来拍板。
