# Git 协作与分支策略

> **结论先行**:团队协作的铁律只有一条——**main 分支永远是能跑的,所有改动都先在自己的分支上做,通过评审再合并回 main**。

## 一、为什么不能都往 main 上推

**结论**:main 是大家共享的"正确版本",直接往上推会互相覆盖、互相踩坏。

**根据**:多人同时改同一个分支,你推一版、他推一版,很快就冲突一片,而且谁都不敢保证 main 还能跑。

**例子**:你在 main 上改一半还没写完,队友拉下来发现编译不过——本该独立的工作互相阻塞了。

## 二、分支策略:一人一条 feature 分支

**结论**:每做一个功能,从 main 切一条分支,做完再合回去。

```
main(始终可运行)
 ├─ feature/login      (你做登录)
 ├─ feature/user-admin (队友做用户管理)
 └─ fix/password-check (另一人修 bug)
```

**根据**:各做各的分支,互不干扰;只有"完成且通过检查"的代码才进 main,main 就始终是干净可跑的。

## 三、标准流程:从开分支到合并

**结论**:六步走完一个功能。

```
1. git checkout -b feature/login   # 从 main 切出新分支
2. 写代码 + 本地跑测试
3. git commit                       # 提交,写清这次改了什么
4. git push 开 Pull Request(PR)    # 推上去,发起合并请求
5. CI 自动跑测试 + 队友/AI 评审       # 见下方代码评审
6. 通过后合并进 main,删掉该分支
```

**例子**:这正是 [AI 工作流](/engineering/ai-workflow) 里"push 后让 Claude 在 PR 里审一轮"的前提——有了 PR,才有地方做评审。

## 四、写好提交信息

**结论**:提交信息写"做了什么、为什么",不写"修改了一下"。

**根据**:半年后回头看历史,清晰的提交信息能让你一眼想起当时改了什么;含糊的等于没写。

**例子**:

- ✅ `修复登录:密码错误时不暴露账号是否存在`
- ❌ `update`、`改了改`、`fix bug`

## 五、保护 main

**结论**:在 GitHub 给 main 开"分支保护"——**禁止直接 push,必须走 PR + 检查通过**。

**根据**:把"必须评审才能合并"变成制度,而不是靠自觉。人会忘,规则不会。

## 本章小结

- **结论**:main 永远可跑,改动走 feature 分支 + PR;
- **根据**:分支隔离避免互相踩坏,PR 提供评审入口;
- **例子**:清晰的提交信息和分支保护,把好习惯变成制度。
