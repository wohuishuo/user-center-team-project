# 4 · 软件实现

> **结论先行**:实现就是把设计变成能跑的代码。写得好不好,看三条——**命名清楚、不重复、先完成再优化**。

## 一、代码首先是写给人看的

**结论**:代码要让**半年后的你和队友**一眼看懂,清楚比聪明重要。

**根据**:代码被读的次数远多于被写的次数。一段炫技但难懂的代码,维护时是负担;一段朴实清晰的代码,谁接手都顺。

**例子**:`getSafetyUser(user)` 一看就知道"拿到脱敏后的用户";`process(u)` 就得点进去猜。命名本身就是注释。

## 二、不重复:重复就提取

**结论**:同一段逻辑出现第二次,就该把它提成一个方法或常量。

**根据**:重复代码意味着改的时候要改好几处,漏一处就出 bug。提取成一处,改一次就够。

**例子**:用户中心里"脱敏"这件事在多个接口都要做,于是统一成 `getSafetyUser()`;角色判断的数字 `1` 提成常量 `ADMIN_ROLE`——这就是[后端架构](/project/backend)里"公共件"的由来。

::: tip 常量别散落成"魔法数字"
代码里突然冒出的 `1`、`"yupi"` 这种没有名字的值叫"魔法数字",别人看不懂含义。提成有名字的常量(`ADMIN_ROLE`、`SALT`),意图就清楚了。
:::

## 三、防御式编程:先想"如果出错"

**结论**:写每个方法时,先想"传进来的东西要是不对怎么办",在最前面挡住。

**根据**:外部传入的参数不可信(可能为空、超长、含特殊字符)。在方法开头先校验、不合法就早早返回或抛异常,后面的逻辑才能放心写。

**例子**:注册方法第一件事就是校验账号密码非空、够长、格式对——不合法直接抛 `BusinessException`,根本走不到后面的入库逻辑(见 [认证模块](/project/auth))。

## 四、先完成,再优化

**结论**:先把功能跑通,再回头优化;不要一开始就追求完美。

**根据**:过早优化容易把简单问题复杂化,还可能优化了根本不重要的地方。先有能用的版本,才知道哪里真的需要优化。

**例子**:用户中心先做出"能注册能登录"的最小版本,跑通后再提取公共方法、加全局异常处理——这个顺序,正是 [软件生命周期](/engineering/lifecycle) 里"先完成再迭代"的体现。

## 五、AI 时代的实现

**结论**:让 [Claude Code](/engineering/ai-workflow) 生成代码,但这三条标准由你来把关。

**根据**:AI 能快速产出代码,但"命名清不清楚、有没有重复、边界处理够不够"仍需要人判断。AI 负责出力,你负责定标准、做审校。

## 本章小结

- **结论**:好实现 = 命名清楚 + 不重复 + 先完成再优化;
- **根据**:代码主要是给人读的,重复和过早优化都是负担;
- **例子**:`getSafetyUser`、`ADMIN_ROLE`、注册开头的参数校验,都是这三条的活例子。
