# 3 · 软件设计(结构化 / 面向对象 / UML)

> **结论先行**:软件设计就一件事——**把"要做什么"(需求)变成"怎么做"(结构)**。不管用结构化还是面向对象方法,目标都是一个:**高内聚、低耦合**。

这一章把课程里的设计基础、结构化方法、面向对象方法和 UML 合在一起讲。它们不是四件事,而是"做设计"这一件事的不同工具。

## 一、好设计的唯一标准:高内聚、低耦合

**结论**:判断设计好坏,记住一句话——**一个模块自己该管的都管好(高内聚),和别人的牵连尽量少(低耦合)**。

**根据**:内聚高,模块职责清晰、好理解;耦合低,改一处不会牵连一片。两者一起,系统才能"局部修改不引发全局崩塌"。

**例子**:用户中心里,密码加密只放在 Service 层一处(高内聚);换加密算法时,Controller 和数据库层完全不用动(低耦合)。这正是 [认证模块](/project/auth) 能从容升级的原因。

::: tip 配套原则:模块化、抽象、信息隐藏
这三个词其实都是"高内聚低耦合"的不同说法——把系统拆成模块、只暴露必要的接口、把实现细节藏起来。
:::

## 二、两种方法,看世界的两种角度

**结论**:结构化方法按"功能/流程"拆,面向对象方法按"对象/职责"拆。

| | 结构化方法 | 面向对象方法 |
| --- | --- | --- |
| 核心问题 | 数据怎么**流动**、被怎么**处理** | 系统里有哪些**对象**、各自负责什么 |
| 主要工具 | 数据流图(DFD)、数据字典 | 类图、领域模型、设计模式 |
| 适合 | 流程清晰的系统 | 对象关系复杂的系统 |
| 课程文档 | 需求/系统设计说明书(结构化) | 需求/系统设计说明书(面向对象) |

**根据**:课程要求同一个项目用两种方法各做一份文档,正是为了让你体会"同一个系统,可以从流程视角看,也可以从对象视角看"。

## 三、结构化:跟着数据流走

**结论**:结构化设计的核心产物是**数据流图**——画出数据从哪来、经过哪些处理、到哪去。

**例子**(用户中心登录的数据流):

```
[用户] --账号密码--> (校验格式) --合法--> (查数据库) --用户记录--> (比对密码)
                                                              |
                                            通过 --> (记录登录态) --脱敏信息--> [用户]
```

每个圆圈是一个"处理",箭头上是"流动的数据"。把它画清楚,后端各层的职责就自然浮现了。

## 四、面向对象:找出对象和职责

**结论**:面向对象设计的核心是**领域模型**——找出系统里的关键对象,以及它们各自的职责。

**例子**(用户中心的领域模型,用类图表示):

```
┌────────────────────┐
│       User         │   ← 对象
├────────────────────┤
│ id, userAccount     │   ← 属性(它有什么)
│ userPassword ...    │
├────────────────────┤
│ register()          │   ← 方法(它能做什么)
│ login()             │
│ getSafetyUser()     │   ← 脱敏:这是 User 自己的职责
└────────────────────┘
```

**根据**:把"脱敏"放进 User 的职责里,而不是散落在各个 Controller,就是面向对象在落实"高内聚"。

## 五、UML:画设计的通用语言

**结论**:UML 是一套画图标准,让团队用同一种符号交流设计,不会各画各的。

**根据**:三种图覆盖了大部分场景——

| 图 | 回答的问题 | 用在哪 |
| --- | --- | --- |
| **用例图** | 谁能用系统做什么 | 需求阶段(见 [需求工程](/theory/02-requirements)) |
| **类图** | 系统里有哪些对象、什么关系 | 面向对象设计 |
| **时序图** | 一次操作中,对象间怎么按顺序调用 | 设计复杂交互 |

**例子**:登录的时序图能清楚画出"Controller → Service → Mapper → 数据库"的调用顺序,一眼看懂分层。

## 本章小结

- **结论**:设计就是把需求变成"高内聚低耦合"的结构;
- **根据**:结构化按流程拆、面向对象按对象拆,UML 是共同的画图语言;
- **例子**:把"脱敏"收进 User 的职责,就是高内聚的活例子。
