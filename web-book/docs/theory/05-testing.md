# 5 · 软件测试

> **结论先行**:测试的目的是**在用户之前发现问题**。记住一条核心原则——**越早测越省**(最左原则),和两种思路——**黑盒看功能、白盒看代码**。

## 一、测试是在"找问题",不是"证明没问题"

**结论**:测试的心态应该是主动找茬,而不是走个过场证明"它能跑"。

**根据**:你只验证正常情况,永远发现不了 bug;bug 都藏在边界和异常里(空值、超长、并发)。带着"我要把它弄崩"的心态去测,才测得出东西。

**例子**:注册功能,正常账号能注册不算测过;要试空账号、3 位短账号、含特殊字符的账号、重复账号——这些边界才是 bug 的藏身处。

## 二、最左原则:越早测越便宜

**结论**:能在早期发现的问题,绝不留到后期。

**根据**:同一个 bug,编码时发现改一行,上线后发现可能是一次事故。测试越往"左"(越早)做,修复成本越低。这和 [软件生命周期](/engineering/lifecycle) 的"早发现早便宜"是同一回事。

```
需求 → 设计 → 编码 → 测试 → 上线
  ↑ 在这里发现错误,改起来最便宜
                      ↑ 拖到这里才发现,代价最大
```

## 三、黑盒与白盒:两种视角

**结论**:黑盒测"功能对不对",白盒测"代码每条路走没走到"。

| | 黑盒测试 | 白盒测试 |
| --- | --- | --- |
| 看什么 | 只看输入输出,不管内部 | 看代码内部的每条分支 |
| 问的问题 | "给这个输入,输出对吗?" | "每个 if 分支都被测到了吗?" |
| 用户中心例子 | 输错密码,会不会拒绝登录? | 注册方法里每条校验分支都被覆盖? |

**根据**:两者互补。黑盒保证"对用户而言功能正确",白盒保证"代码没有没测到的死角"。

## 四、测试的三个层次

**结论**:从小到大分三层——单元、集成、系统。

- **单元测试**:测单个方法(`userRegister` 对短密码报错);
- **集成测试**:测多个组件配合(注册真的写进了数据库);
- **系统测试**:测整条用户流程(打开页面→注册→登录→进首页)。

**例子**:这三层在用户中心里怎么落地、用什么工具(JUnit 5、Testcontainers、Vitest),见实战篇的 [测试](/project/testing) 一章。

## 五、需求、测试、文档三者呼应

**结论**:每条需求对应一条测试,测试结果汇成测试报告。

**根据**:需求"密码 ≥8 位" → 测试"输入 7 位应被拒绝" → 报告记录"该用例通过"。三者一一对应,质量才可追溯。

## 本章小结

- **结论**:测试是在用户之前主动找问题;
- **根据**:最左原则让修复最便宜,黑盒白盒互补;
- **例子**:注册的边界用例(空/短/特殊字符/重复)正是该重点测的地方。
