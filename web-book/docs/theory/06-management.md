# 6 · 软件维护与项目管理

> **结论先行**:软件上线不是终点,而是**维护的起点**;而项目能不能按时做完,靠的是**管好进度、分工、风险**这三件事。这一章讲"交付后怎么活下去"和"团队怎么把项目推完"。

## 一、维护:软件一生的大头

**结论**:维护占软件整个生命周期成本的一大半,分三类。

| 维护类型 | 干什么 | 用户中心例子 |
| --- | --- | --- |
| **纠错性** | 修上线后发现的 bug | 修复登录偶发报错 |
| **适应性** | 跟上环境变化 | MySQL 升级、依赖升版 |
| **完善性** | 加新功能、优化 | 加"修改用户信息"功能 |

**根据**:软件不是交付完就不动了——需求会变、环境会变、bug 会冒出来。所以"好不好维护"必须在设计时就考虑,这正是我们一直强调[高内聚低耦合](/theory/03-design)的原因。

**例子**:用户中心因为分了层,后来要加"用户信息修改",只需在 Service 加方法、Controller 加接口,不用动现有的登录逻辑——这就是为维护而设计的回报。

## 二、技术债:欠的早晚要还

**结论**:为赶进度而走的捷径,会变成"技术债",越拖利息越高。

**根据**:临时硬编码、跳过测试、复制粘贴——当时省了时间,但后面每次改动都要为它多付代价。债不是不能欠,而是要**记得还**。

**例子**:用户中心初期把校验码功能标成 `// todo`,这是有意识地欠债;但要记在清单里,迭代时补上,而不是假装它不存在。

## 三、项目管理:管好三件事

**结论**:项目管理的核心是管**进度、分工、风险**。

**进度**:把大目标拆成小任务,排出顺序和时间。

**分工**:每个人认领明确的模块,边界清楚(对应课程要交的"项目分工表")。

```
组长:统筹 + 需求/设计文档
成员A:后端认证模块 + 测试
成员B:前端页面 + 联调
成员C:数据库设计 + 部署
```

**风险**:提前想"哪里可能出问题",并准备对策。

| 风险 | 对策 |
| --- | --- |
| 某人进度卡住 | 任务可拆、可支援,不要一个人扛关键路径 |
| 临近 deadline 才合并 | 早合并、勤合并,走 [PR 流程](/engineering/git-workflow) |
| 上线才发现 bug | 提前写[测试](/project/testing)、做[评审](/engineering/code-review) |

**根据**:项目失败很少是因为技术不行,多是进度失控、分工不清、风险没预案。管好这三件事,比多写两行代码更能决定成败。

## 四、AI 时代的管理

**结论**:AI 改变了"谁出力",但进度、分工、风险仍要人来管。

**例子**:用 AI 加速编码,省下的时间该投到设计、评审和沟通上——这些"人和人之间"的事,AI 替不了。

## 本章小结

- **结论**:维护是软件一生的大头,项目靠管进度/分工/风险推完;
- **根据**:为维护而设计(高内聚低耦合)、有意识地还技术债;
- **例子**:用户中心分层让加功能不动旧逻辑,分工表让每人边界清楚。
