# 1 · 软件工程概述

> **结论先行**:软件工程,就是**用工程的纪律来做软件**——按流程、讲质量、能协作,而不是一个人凭感觉乱写。它存在的理由,是为了治"软件危机"。

## 一、什么是软件危机

**结论**:软件危机指的是——**软件越做越大,但做不完、做不对、没法维护**。

**根据**:程序规模一大,光靠"聪明 + 加班"就压不住了:需求理解错、进度拖延、上线一堆 bug、改一处坏一片。这些不是个人能力问题,是缺方法的问题。

**例子**:一个人写几百行小程序没问题;但十个人合作做几万行的用户中心,如果没有统一流程和分工,很快就乱成一团。

## 二、软件工程怎么治它

**结论**:用三样东西治软件危机——**流程、质量、协作**。

| 药方 | 解决什么 | 在本书哪里 |
| --- | --- | --- |
| **流程** | 乱做、漏步骤 | [软件生命周期](/engineering/lifecycle) |
| **质量** | bug、不安全 | [软件测试](/theory/05-testing) · [代码评审](/engineering/code-review) |
| **协作** | 多人踩踏、各写各的 | [Git 协作](/engineering/git-workflow) · [项目管理](/theory/06-management) |

**根据**:软件危机的本质是"复杂度失控"。流程让步骤可控,质量手段让缺陷可控,协作机制让团队可控——三管齐下,复杂项目才做得动。

## 三、软件 ≠ 代码

**结论**:软件不只是代码,还包括**文档、数据和让它能运行/维护的一切**。

**根据**:代码只是软件的一部分。需求文档、设计说明、数据库、部署配置、测试用例……缺了这些,代码没人看得懂、也没法维护。这正是课程要交那么多文档的原因。

**例子**:用户中心交付的不只是能跑的程序,还有需求规格、系统设计、数据设计、测试报告——它们合起来才是完整的"软件"。

## 四、用工程思维看用户中心

**结论**:这本书就是软件工程思维的一次完整演练。

**根据**:我们对用户中心做的每件事,都对应一条工程纪律——

- 先分析[需求](/project/requirements),不上来就写代码;
- 先做[设计](/theory/03-design)和[数据库](/project/database),再编码;
- 写[测试](/project/testing)守质量;
- 走[评审和分支](/engineering/git-workflow)做协作;
- 用 [Docker + CI/CD](/project/deployment) 规范部署。

**例子**:同样做一个登录功能,"凭感觉写"和"按工程做"的区别,就是上线后睡得着觉和睡不着觉的区别。

## 本章小结

- **结论**:软件工程 = 用工程纪律(流程/质量/协作)做软件;
- **根据**:它治的是软件危机,即复杂度失控;
- **例子**:软件不只是代码,用户中心的文档、数据、部署同样是交付物。
