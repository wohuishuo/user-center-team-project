# 第2章 软件需求工程

> ★ 完整样板章节。理论 + 用户中心案例 + 课程文档对照,一应俱全。其余理论章节请照这个结构写。

## 这一章在讲什么

软件做坏掉,**最贵的错误往往发生在"还没写一行代码"的需求阶段**。需求工程就是系统地搞清楚:**到底要做什么、为谁做、做到什么程度**——并把它写下来,让所有人对齐。

需求工程一般分四步:

```
需求获取 → 需求分析 → 需求规格说明 → 需求验证
  (问)      (想)         (写)          (核对)
```

## 一、需求获取:需求从哪来

需求不会自己冒出来,要主动去"挖"。来源通常有:

- **客户 / 老板**:他们想要什么业务效果;
- **产品经理**:把业务翻译成功能;
- **最终用户**:实际使用中的痛点;
- **开发者自己**:技术上必须有的东西(如日志、监控)。

::: tip 关键意识:需求要"评估",不是照单全收
不是所有提出来的需求都该做。要权衡 **实现成本 vs 业务价值**。笔记里有个例子:花 10 天只满足 1 个用户的不合理需求,就不值得做。**学会说"不"也是需求工程的一部分。**
:::

### 用户中心的需求是怎么来的

用户中心的出发点是一个**真实痛点**:一家公司有多个系统,用户要在每个系统重复注册、重复登录,体验差、数据也分散。

> 类比:QQ 号能登录腾讯全家桶,支付宝账号能登录阿里全家桶。把"用户"这件事**抽出来做成独立服务**,所有业务系统都来对接它。

由这个痛点,推导出三组功能需求(见下)。

## 二、需求分析:功能性 vs 非功能性

需求分两类,**两类都要写,新手常常漏掉第二类**:

### 功能性需求(系统要"做"什么)

用户中心的功能性需求:

| 编号 | 功能 | 谁能用 | 说明 |
| --- | --- | --- | --- |
| F1 | 注册 | 所有人 | 账号 + 密码 + 校验码注册 |
| F2 | 登录 / 登出 | 所有人 | 登录后保持登录态 |
| F3 | 用户管理 | 仅管理员 | 查询、删除用户 |
| F4 | 用户校验 | 仅内部用户 | 用"星球编号"校验身份 |

### 非功能性需求(系统要"多好")

这些不直接对应某个按钮,但**决定系统能不能上线**:

| 类别 | 用户中心的要求 |
| --- | --- |
| **安全性** | 密码必须加密存储(BCrypt);接口必须鉴权;防越权访问 |
| **性能** | 登录响应 < 500ms;支持后续水平扩展 |
| **可用性** | 错误提示清晰;表单即时校验 |
| **可维护性** | 分层架构;统一返回格式与异常处理 |

::: warning 课程文档高频扣分点
**数据安全**就是典型的非功能性需求,很多组只写功能、不写安全就被扣分。要针对"不同访问者 / 访问类型 / 数据对象"分别说明:谁能查看、谁能获取、传输怎么保护。详见 [文档工坊](/docs-workshop/)。
:::

## 三、需求规格说明:用"用例"写下来

把需求**结构化地写下来**,常用工具是**用例(Use Case)**。一个用例描述"某个角色为达成某个目标,和系统的一次完整交互"。

### 用例图(用户中心)

```
        ┌──────────────────────────────────┐
        │           用户中心系统            │
        │                                  │
 普通用户 ─┼──→ (注册)                       │
        │   ─→ (登录)                       │
        │   ─→ (登出)                       │
        │                                  │
 管理员  ─┼──→ (查询用户)  «include» → (登录) │
        │   ─→ (删除用户)                   │
        └──────────────────────────────────┘
```

### 用例规约示例:用户登录

把关键用例写成**结构化规约**,这是需求规格说明书的核心内容:

| 项 | 内容 |
| --- | --- |
| **用例名** | 用户登录 |
| **主角色** | 普通用户 |
| **前置条件** | 用户已注册 |
| **主成功流程** | 1. 用户输入账号、密码<br>2. 系统校验格式(账号 ≥4 位、密码 ≥8 位、无特殊字符)<br>3. 系统比对密码<br>4. 校验通过,记录登录态<br>5. 返回脱敏后的用户信息 |
| **扩展流程** | 2a. 格式不合法 → 提示并终止<br>3a. 密码错误 → 提示"账号或密码错误"(不暴露是哪个错) |
| **后置条件** | 用户处于已登录状态 |

> 这个用例直接对应 [实战篇 · 认证模块](/project/auth) 的代码实现——理论和代码是同一件事的两面。

## 四、需求验证:确认没写错、没写漏

需求写完要**回头核对**:

- **完整性**:每个功能都有对应需求?边界情况(密码错、账号重复)都覆盖了?
- **一致性**:有没有自相矛盾的地方?
- **可验证性**:每条需求都能被测试验证?("系统要快"不可验证,"登录 <500ms"可验证)
- **可追溯性**:每条需求能追到来源,也能追到后面的设计与测试。

::: tip 与测试的呼应
好的需求天然能转成测试用例。"密码 ≥8 位"这条需求,直接对应一条单元测试:输入 7 位密码应被拒绝。需求 → 测试的可追溯性,见 [第6章 软件测试](/theory/06-testing)。
:::

## 本章小结

- 需求工程四步:**获取 → 分析 → 规格说明 → 验证**;
- 需求要**评估价值**,不是照单全收;
- **功能性 + 非功能性都要写**,安全是最易漏的非功能需求;
- 用**用例**把需求结构化写下来,它直接对应后面的设计、代码和测试。
