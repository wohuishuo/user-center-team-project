# 需求分析

> **结论先行**:用户中心要做的事,一句话说清——**把"用户"从各个业务系统里抽出来,做成一个独立的登录与管理服务**。本章把这句话拆成可实现的功能清单、用例和边界。

这一章是 [理论篇 · 需求工程](/theory/02-requirements) 在项目里的落地。理论讲"怎么分析需求",这里讲"用户中心的需求具体是什么"。

## 一、为什么要做用户中心

**结论**:因为"用户"是多个系统都要用、却被重复造的轮子。

**根据**:一家公司往往有好几个系统(论坛、面试、课程……),如果每个都自己做一套注册登录,用户要重复注册、数据散落各处、维护成本翻倍。

**例子**:QQ 号能登录腾讯全家桶,支付宝账号能登录阿里全家桶。把"用户"抽成独立服务,所有业务系统都来对接它——这就是用户中心。

## 二、功能性需求:系统要做什么

**结论**:用户中心只做四组功能,且每组都有明确的"谁能用"。

| 编号 | 功能 | 谁能用 | 说明 |
| --- | --- | --- | --- |
| F1 | 注册 | 所有人 | 账号 + 密码 + 校验 |
| F2 | 登录 / 登出 | 所有人 | 登录后保持登录态 |
| F3 | 用户管理 | 仅管理员 | 查询、删除用户 |
| F4 | 用户校验 | 仅内部用户 | 用"星球编号"等标识校验身份 |

**根据**:功能少而清晰,才能先做出能用的版本,再慢慢加。贪多求全是新手项目最常见的坑。

::: tip 边界:不做什么也要写清
用户管理**只做查询和删除**,初期不做"后台新增/修改用户"。明确"不做什么",和明确"做什么"同样重要——它防止范围无限膨胀。
:::

## 三、非功能性需求:系统要多好

**结论**:功能之外,这四条决定用户中心能不能真正上线。

| 类别 | 要求 |
| --- | --- |
| **安全性** | 密码加密存储(BCrypt);接口必须鉴权;防越权访问 |
| **性能** | 登录响应快;支持后续多机水平扩展 |
| **可用性** | 错误提示清晰;表单即时校验 |
| **可维护性** | 分层架构;统一返回格式与异常处理 |

**根据**:很多团队只写功能、漏掉非功能需求,结果做出来的东西"能跑但不能用"——比如密码明文存储,一上线就是事故。安全是这里最不能省的一条。

## 四、核心用例:用户登录

**结论**:把关键功能写成**结构化用例**,需求才算真正"说清楚了"。

| 项 | 内容 |
| --- | --- |
| **用例名** | 用户登录 |
| **主角色** | 普通用户 |
| **前置条件** | 用户已注册 |
| **主成功流程** | 1. 输入账号、密码<br>2. 系统校验格式(账号 ≥4 位、密码 ≥8 位、无特殊字符)<br>3. 系统比对密码<br>4. 通过则记录登录态<br>5. 返回脱敏用户信息 |
| **扩展流程** | 2a. 格式不合法 → 提示并终止<br>3a. 密码错误 → 提示"账号或密码错误"(不暴露是哪个错) |
| **后置条件** | 用户处于已登录状态 |

**例子**:这个用例直接对应 [认证模块](/project/auth) 的代码——需求里写的每一步校验,代码里都有一行对应。需求和实现是同一件事的两面。

## 五、需求验证:回头自检

**结论**:需求写完,用四个问题过一遍。

- **完整**:每个功能都有需求?边界(密码错、账号重复)都覆盖了?
- **一致**:有没有自相矛盾?
- **可验证**:每条都能被测试?("要快"不可验证,"登录响应快"也要给个具体目标)
- **可追溯**:每条需求能追到来源、也能追到后面的设计与测试?

## 本章小结

- **结论**:用户中心 = 抽出来的独立用户服务,只做四组功能;
- **根据**:功能少而清晰才能先跑起来;安全是最不能省的非功能需求;
- **例子**:登录用例的每一步,都对应认证模块的一行代码。
